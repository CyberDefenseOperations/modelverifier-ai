Integration Guide

The AI Model & System Assurance Control Matrix is available as a CORS-enabled JSON API — fetch from any origin, no key required. 54 controls, 8 framework mappings, 11 deployment profiles, machine-readable obligation objects, and structured implementation guidance.

What this is
This matrix verifies whether a specific AI system in a specific use case, under a specific obligation set, has enough structured evidence to be trusted. It is not a model inventory checklist and not a runtime interceptor. It documents what must be evaluated, by which test, with what evidence, at what confidence, for which actor.
The governed object is the assurance target — not just the model artifact. The same base model can be low-risk in an internal summarization tool and high-risk in employment screening. Classification, obligations, and required controls are all derived from assurance target properties, not fixed model attributes.

01 Quick start

One fetch — no API key, no SDK, no rate limit.

// Browser or Node 18+
const { dataset } = await fetch(
  'https://modelverifier.ai/integration/model-controls-full.json'
).then(r => r.json());

dataset.meta.control_count    // 54
dataset.meta.version          // "1.0.0"
dataset.controls.length       // 54
dataset.profiles.length       // 11
# curl
curl https://modelverifier.ai/integration/model-controls-full.json \
  | jq '.dataset.meta'

02 Dataset shape & build warnings

Everything is under dataset. The top-level keys:

KeyTypeDescription
metaobjectBuild metadata: version, hash, counts, generated timestamp
controls[]array54 control records sorted LI→CR then by number
references[]arrayDeduplicated source documents cited by controls
layers[]array6 layer definitions with control ID lists
planes[]array4 assurance plane definitions
profiles[]array11 deployment profiles with required/recommended control IDs
patterns[]arrayImplementation pattern summaries per control
framework_coverage[]arrayPer-framework mapping statistics
capability_coverage[]arrayControls indexed by capability level and domain
lifecycle[]arrayOrdered 6-stage model assurance lifecycle

Each control has a stable id (LI-01 through CR-08) that is never reused. The canonical_id is the globally unique cross-domain identifier: apeiris://model/controls/LI-01.

Build warnings: The current dataset sets dataset.meta.has_warnings: true with warning_count: 1 but does not yet emit a structured warnings[] array. API consumers should check has_warnings and treat it as a signal to consult the release manifest for details. A typed warnings[] block (severity, object_path, message, remediation, blocks_release) is on the v1.1 data schema roadmap.

GPAI profiles: EU AI Act Chapter V and VI impose distinct obligations on providers of general-purpose AI models that are separate from, and additive to, high-risk system obligations. A frontier model provider, a downstream enterprise deployer, and an orchestrated-service provider do not have the same duties. The gpai-provider and gpai-systemic-risk profiles are now live — both profiles became applicable 2025-08-02 (12 months after EU AI Act entry into force). The systemic risk profile (Ch. VI) is additive to the GPAI provider profile (Ch. V) — both are simultaneously active for providers above the 10²⁵ FLOPs threshold or designated by the AI Office. 
// Get all required controls for a profile + the baseline
function requiredControls(dataset, profileId) {
  const profile = dataset.profiles.find(p => p.profile_id === profileId);
  const ids = new Set([
    ...dataset.meta.baseline_controls,
    ...(profile?.required_controls ?? []),
  ]);
  return dataset.controls.filter(c => ids.has(c.id));
}

const euRequired = requiredControls(dataset, 'eu-high-risk');
// Returns baseline (15) + EU High-Risk required controls
// Controls required OR recommended for a profile
function applicableControls(dataset, profileId) {
  const profile = dataset.profiles.find(p => p.profile_id === profileId);
  const req = new Set(profile?.required_controls ?? []);
  const rec = new Set(profile?.recommended_controls ?? []);
  const base = new Set(dataset.meta.baseline_controls);
  return dataset.controls.map(c => ({
    ...c,
    applicability: base.has(c.id) ? 'baseline'
      : req.has(c.id)  ? 'required'
      : rec.has(c.id)  ? 'recommended'
      : 'not-applicable'
  })).filter(c => c.applicability !== 'not-applicable');
}

AICM v1.1 (June 22, 2026): The framework catalog is updated to CSA AICM v1.1 (247 control objectives, 18 domains). Existing control-level AICM mappings were authored against v1.0.3 IDs — validate these against the v1.1 document before relying on specific AICM control IDs in downstream tooling. New domains from the 8-domain expansion require ID mapping validation in the next authoring cycle.

NIST AI 600-1: Currently implied through nist_rmf citations. The GenAI Profile has distinct coverage (confabulation, CBRN uplift, IP, compute) that doesn't map cleanly to base RMF subcategories. It will be split into a separate nist_ai_600_1 key in v1.1, primarily affecting EV and BH layer controls.

SR 26-2 scope: This is supervisory guidance, not binding law. It does not set enforceable standards or prescriptive requirements; non-compliance will not result in supervisory criticism on its own. It applies to supervised banking organizations; heightened expectations apply based on organization size, complexity, and model risk profile — not a hard $30B asset threshold.

07 Cross-domain navigation

Controls reference related controls in other Apeiris domains via cross_domain.references[] using stable apeiris:// URIs. These URIs are permanent and never reused.

apeiris://model/controls/LI-01 → this domain (modelverifier.ai)
apeiris://security/controls/IA-02 → securitycontrols.ai
apeiris://privacy/controls/DP-03 → (future privacy domain)
apeiris://compliance/controls/RF-01 → (future compliance domain) 
// Resolve an apeiris:// URI to find the domain
function parseApeiriUri(uri) {
  const m = uri.match(/^apeiris:\/\/([^/]+)\/controls\/([A-Z]{2}-\d{2})$/);
  if (!m) return null;
  return { domain: m[1], id: m[2] };
}

// Cross-domain links for a control
const li01 = dataset.controls.find(c => c.id === 'LI-01');
const refs = li01.cross_domain?.references ?? [];
refs.forEach(uri => {
  const loc = parseApeiriUri(uri);
  console.log(`Related: ${loc?.id} in ${loc?.domain}`);
});
Namespace registry — The global prefix registry lives in apeiris-control-core/namespace-registry.json. model domain owns prefixes LI, TG, EV, OA, BH, CR. security domain owns IA, EC, PT, GV, RT, AS. Short IDs are display-only; canonical IDs are the durable reference.

08 Scope boundaries — what this covers and doesn't

This matrix is a knowledge domain and evidence framework, not a runtime interceptor. It documents what must be evaluated, by which method, with what evidence — it does not execute the evaluation. The Apeiris platform executes against these controls; the controls define what counts as evidence.

This matrix owns
  • Model/system fitness, evaluation, and validation
  • Pre-release evaluation gates and regression testing
  • Behavioral measurement inside workflows (tool-call quality, output grounding)
  • Training data governance and poisoning prevention
  • Production monitoring, drift detection, behavioral boundaries
  • Incident response and corrective action
  • Framework mapping — what each control supports
  • Evidence production — what artifacts count as proof
  • Maturity — where industry is and what it takes to advance
Other Apeiris verifiers own
  • securitycontrols.ai — artifact integrity, runtime isolation, prompt-injection enforcement, output filtering, vulnerability response
  • Privacy verifier — lawful data use, retention, data-subject rights, minimization
  • Compliance verifier — obligation interpretation, legal applicability, jurisdiction
  • Agentic Control — delegated authority, tool authorization, workflow enforcement
Why this distinction matters for integration: If you find a concern not covered here, it is likely owned by an adjacent verifier. For example: "does the model resist prompt injection?" — securitycontrols.ai owns the enforcement; this matrix owns the measurement of injection resistance (EV-04). The evidence from both verifiers flows into the same assurance target via apeiris:// URIs, so they compose without duplication.

What this matrix does not do: It does not intercept model outputs in real time, run a Z3 theorem prover, host an adversarial red-teaming pipeline, or enforce access controls. Those are Apeiris platform capabilities. The controls document what those capabilities must verify and what evidence they must produce — that is the contract between the matrix and the platform.

The shared evidence architecture uses apeiris:// URIs so one verifier's evidence artifacts can be consumed by another without re-running the same control.

09 Release manifest verification

// Verify dataset integrity
const manifest = await fetch(
  'https://modelverifier.ai/integration/release-manifest.json'
).then(r => r.json());

// Compare against dataset.meta.content_hash
const storedHash    = manifest.artifacts[0].content_hash;
const datasetHash   = dataset.meta.content_hash;
console.log('Hash match:', storedHash === datasetHash);

The manifest is signed on every build. The content_hash covers the full dataset object before the meta timestamp is applied — it will match even if you fetch at a different time.
